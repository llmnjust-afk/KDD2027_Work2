\section{Experiments and Results}
\label{sec:exp}

\subsection{Experimental Setup}
\label{sec:setup}

\textbf{Models.} We evaluate five frontier LLMs via an OpenAI-compatible endpoint (ChatAnywhere): GPT-4o, GPT-4o-mini, DeepSeek-V3 (\texttt{deepseek-chat}), DeepSeek-R1 (\texttt{deepseek-reasoner}), and Qwen3-max (\texttt{qwen3-max-2026-01-23}). Note that \texttt{qwen3-max-2026-01-23} is a pinned version snapshot, while the other four are rolling aliases that may be updated by the provider. All models use temperature 0 and a maximum of 6--8 ReAct steps. API calls were made during June--July 2026.

\textbf{Task suites.} We use three complementary task suites (\Cref{tab:task-summary}):

\input{tables/table_task_summary}

\textbf{Protocol and scale.} Each (model, task) pair is run for 3 repetitions at temperature 0. In total we analyse \textbf{5{,}984 agent runs} across \textbf{318 unique tasks} (95 synthetic $+$ 23 UCI $+$ 200 DSBench), yielding \textbf{3{,}940 failures}. Of these, \textbf{2{,}218 (38.1\%) are infrastructure failures}: 1{,}273 are API errors (status$=$ERR, the request failed) and 945 are unknown infrastructure issues (status$=$OK but tokens$=$0, indicating empty or unparseable responses). We acknowledge that the current logging does not distinguish between parse failures, timeouts, rate limits, and format incompatibilities within the ``unknown'' category; future versions of the framework will log these subtypes separately. This leaves \textbf{3{,}602 valid runs} with automatically generated diagnostic labels. The 47-trace human annotation protocol (\Cref{app:annotation}) provides gold-label validation on a stratified subset. All tables are auto-generated from a single source-of-truth manifest (\texttt{manifest\_v3.csv}, 32 fields) with assertion checks\footnote{See \texttt{build\_manifest\_and\_tables.py} in our supplementary code.}.

\textbf{Repetition agreement.} Under temperature 0, 89.9\% of (task, model, variant) groups show full agreement across 3 repetitions (all correct or all incorrect). The 10.1\% disagreement rate reflects server-side non-determinism in the API endpoint, not sampling randomness. DeepSeek-R1 shows 100\% agreement (all infrastructure failures), while GPT-4o-mini shows 84.6\%.

\textbf{Statistical tests.} Verifier comparisons use the McNemar test (appropriate for paired binary outcomes), reporting $b$ (baseline correct \& verifier wrong), $c$ (baseline wrong \& verifier correct), and the $\chi^2$ $p$-value.

\Cref{tab:comprehensive} presents the comprehensive baseline results across all five models, three task suites, and nine diagnostic metrics; it is the single source of truth for all success-rate and SFR figures reported in this paper, and every cell is computed over valid runs only (infrastructure failures excluded, marked ``---'' where no valid runs exist). Three patterns emerge. First, \textbf{DeepSeek-V3 is the strongest model} on both synthetic (88.4\% SR) and UCI (94.2\% SR) tasks, achieving perfect code-execution success on synthetic traps. Second, \textbf{the silent-failure rate is strongly task-dependent}: on synthetic trap tasks, SFR ranges from 39\% to 100\% (failures are often wrong answers, not crashes); on UCI real data, SFR is bimodal---either 0\% (GPT-4o-mini, DeepSeek-V3) or 100\% (GPT-4o); on DSBench, only GPT-4o produces silent failures (38.3\%). Third, \textbf{DSBench is extremely challenging} for all models (0--3.3\% SR), with GPT-4o achieving the highest success rate and the only non-zero SFR, indicating that even the strongest model produces plausible-but-wrong answers on complex financial data. We deliberately report SFR only for model--suite cells that have valid runs; cells consisting entirely of infrastructure failures are left undefined rather than reported as 0\% or 100\%.

\subsection{Result 3: Taxonomy Validation}
\label{sec:res-taxonomy}

We validate the five-stage taxonomy via human annotation. Three independent annotators (blind to system labels) labelled 47 failed traces with (stage, is\_silent, subcategory). Inter-annotator agreement is strong: Fleiss' $\kappa = 0.860$ for stage and $0.881$ for is\_silent, with 89.4\% full agreement across all three annotators. Pairwise Cohen's $\kappa$ values are 0.798, 1.000, and 0.798 (average 0.865). The main source of disagreement is the boundary between \texttt{output\_mismatch} and \texttt{answer\_error} (5 traces).

Against the adjudicated gold labels (majority vote), the rule-based classifier achieves 71.8\% accuracy (Cohen's $\kappa = 0.438$). The confusion matrix reveals a systematic bias: the classifier labels 11 traces as \texttt{output\_mismatch} that humans classify as \texttt{runtime} (precision 1.00, recall 0.52 for output\_mismatch; precision 0.59, recall 1.00 for runtime). This suggests the classifier over-assigns silent failures when the trace contains both execution errors and answer mismatches. We release the individual annotator labels and gold labels with the dataset.

\subsection{Result 4: Zero Self-Correction}
\label{sec:res-recovery}

Across all 5{,}984 runs and all five models, the recovery rate is 0\%: no agent ever corrected a silent failure without external intervention. We note that this is partly a consequence of the experimental protocol: silent failures produce no error signal, so the agent has no opportunity to detect and retry. Our recovery benchmark (\Cref{sec:res-recovery-bench}) addresses this by testing feedback signals.

This zero-recovery finding implies that in the current setting without explicit correctness feedback or internal validation protocol, silent failures will not trigger automatic repair. In a multi-step pipeline, a silent failure at step $k$ propagates to all downstream steps.

\subsection{Result 5: Propagation Depth Analysis}
\label{sec:res-propagation}

\Cref{sec:propagation} defines propagation depth. Here we analyse propagation depth \emph{by model}, revealing a striking pattern (\Cref{fig:propdepth-model}). GPT-4o has a mean propagation depth of 1.10 steps with 24.8\% of its failures reaching long-horizon status (depth $\geq$ 2)---far higher than any other model. GPT-4o-mini has a mean depth of 0.25 (7.1\% long-horizon), while DeepSeek-V3, DeepSeek-R1, and Qwen3-max all have zero propagation depth.

\begin{figure}[t]
\centering
\includegraphics[width=0.95\linewidth]{figures/propdepth_by_model.pdf}
\caption{Propagation depth by model (baseline, valid runs only). GPT-4o exhibits the longest failure propagation (mean 1.10, 24.8\% long-horizon). Other models show depth 0, which may reflect either immediate error detection or early trace termination due to infrastructure failures (see \Cref{tab:comprehensive}).}
\label{fig:propdepth-model}
\end{figure}

We caution that zero propagation depth has two possible interpretations: (1) the failure was immediately detected by the agent (a loud failure at depth 0), or (2) the trace terminated early due to infrastructure failures (API errors, parse failures, or empty responses), leaving no steps to propagate through. Our enriched manifest (\texttt{manifest\_v3.csv}) distinguishes these cases via the \texttt{is\_infra\_failure} field. Among valid (non-infrastructure) failures, GPT-4o's longer propagation reflects its tendency to continue reasoning on a wrong intermediate result rather than crashing, making the failure harder to localise. In contrast, DeepSeek-V3's valid failures are concentrated at depth 0, meaning the error manifests in the final answer without intermediate propagation.

\subsection{Result 6: Oracle-Guided Repair}
\label{sec:res-causal}

\input{tables/table_oracle_repair}

Oracle-guided repair succeeds on 66\% of failed tasks where ground-truth code is available (\Cref{tab:oracle-repair}). This demonstrates \emph{repairability}, not causal attribution: as the negative control in \Cref{sec:disc-negative-control} shows, injecting the oracle solution produces identical results regardless of which step is replaced, because the ground-truth code independently computes the correct answer. We therefore report oracle repair as an upper bound on reparability rather than as evidence of step-level causal localisation. A full causal analysis with necessity/sufficiency testing is future work.

\subsection{Result 7: Verifier Ablation}
\label{sec:res-verifier}

\input{tables/table_verifier}

\Cref{tab:verifier} reports the McNemar test results. The verifier significantly helps Qwen3-max ($+24.9\%$, $p<0.001$) and GPT-4o ($+19.0\%$, $p<0.001$), but is not significant for GPT-4o-mini, DeepSeek-V3, or DeepSeek-R1. Note that GPT-4o's improvement was not detected by the paired $t$-test in our earlier analysis, highlighting the importance of using the correct statistical test for binary outcomes. The verifier's benefit to Qwen3-max may partly reflect format/protocol fix-up rather than reasoning correction, since Qwen3-max's low baseline performance is partly due to ReAct parser incompatibility.

A key finding is that the verifier's effect is \emph{model-dependent}: it helps models with low-to-moderate baseline performance (Qwen3-max at 18.2\%, GPT-4o at 62.1\%) but not the strongest model (DeepSeek-V3 at 88.4\%) or the weakest (DeepSeek-R1 at 0\%). This suggests that rule-based verification captures a ``middle band'' of errors---those where the model can reason about the problem but benefits from a structured check---but cannot rescue fundamentally broken traces (R1) or improve near-perfect ones (V3).

\subsection{Result 10: Recovery Benchmark}
\label{sec:res-recovery-bench}

To test whether feedback signals enable agents to recover from failures, we run a recovery benchmark with 6 conditions on 100 failed traces (80 silent + 20 loud) sampled from synthetic tasks:

\begin{itemize}[leftmargin=*,itemsep=1pt]
\item \textbf{C1 (No feedback):} Re-run the task with no additional signal.
\item \textbf{C2 (Generic feedback):} ``Your previous answer may be incorrect. Please retry.''
\item \textbf{C3 (Logic hint):} ``Check your logic carefully (e.g., count vs sum, groupby vs filter).''
\item \textbf{C4 (Verification checklist):} ``Verify your result by checking data types and aggregation function.''
\item \textbf{C5 (Critic feedback):} GPT-4o-mini reviews the answer and provides critique.
\item \textbf{C6 (Independent rerun):} ``Solve using a completely different approach.''
\end{itemize}

The overall recovery rate is \textbf{15.7\%} (94/600, 95\% bootstrap CI [12.9, 18.5]\%), a substantial improvement over the 0\% recovery in the standard protocol. We analyse recovery at three levels: \emph{attempt} (the agent produces any answer), \emph{detection} (the agent produces a different answer than the original failed attempt, indicating it recognised a potential error), and \emph{repair} (the new answer is correct). The attempt rate is high (97--100\% across conditions), indicating that the agent always tries to respond. The detection rate (producing a different answer) varies by condition, and the repair rate is 15.7\% overall. Critically, \textbf{loud failures recover at 36.7\% (44/120, 95\% CI [27.5, 45.0]\%) while silent failures recover at only 10.4\% (50/480, 95\% CI [7.7, 13.1]\%)}; the difference is highly significant ($\chi^2 = 48.1$, $p < 0.000001$), confirming that silent failures remain far harder to correct even with explicit feedback. No feedback condition significantly outperforms no-feedback (McNemar $p > 0.05$ for all conditions; per-condition 95\% CIs overlap substantially: C1 [10.0, 24.0]\%, C2 [10.0, 25.0]\%, C3 [10.0, 25.0]\%, C4 [8.0, 21.0]\%, C5 [9.0, 22.0]\%, C6 [8.0, 21.0]\%), suggesting that the mere act of re-running with any prompt-level signal triggers partial recovery, but no single feedback type is dramatically more effective.

\subsection{Result 11: Benchmark Tasks and Baselines}
\label{sec:res-benchmark}

We define six potential benchmark tasks. Tasks B--D are the core diagnostic tasks; Tasks A, E, F are auxiliary tasks supported by the framework but not the primary focus:

\begin{itemize}[leftmargin=*,itemsep=1pt]
\item \textbf{Task A (Data-science task solving):} Given data and a question, produce the correct answer. This is the standard task; we report SR as the baseline metric.
\item \textbf{Task B (Observability classification):} Given a trace, predict success / loud / silent. Metric: accuracy and macro-F1.
\item \textbf{Task C (Failure-stage classification):} Given a failed trace, classify the failure stage. The five-stage taxonomy (\Cref{sec:taxonomy}) defines the full class hierarchy. In the current data, four of five stages are instantiated: \texttt{runtime}, \texttt{output\_mismatch}, \texttt{answer\_error}, and \texttt{analytical\_plan}. Code-Generation has zero cases---confirmed by both the v2 rule-based classifier (which compares agent code to reference solutions via \texttt{gt\_path}) and three independent human annotators on 47 traces. This is a genuine finding: current LLM agents do not make operation-level code errors (e.g., using \texttt{.count()} where \texttt{.sum()} is needed) on these tasks; they either solve them, crash, or produce wrong answers for other reasons. The task is therefore instantiated as a \textbf{four-class classification} (runtime vs.\ output\_mismatch vs.\ answer\_error vs.\ analytical\_plan), which captures the silent/loud diagnostic distinction plus approach-level vs.\ extraction-level errors within silent failures.
\item \textbf{Task D (Originating-step localization):} Predict the first erroneous step. Metric: step-level accuracy and mean absolute error.
\item \textbf{Task E (Verifier evaluation):} Given a trace without ground truth, detect whether the answer is correct. Metric: precision, recall, F1. Supported by the verifier ablation (\Cref{sec:res-verifier}).
\item \textbf{Task F (Repair):} Given a failed trace, generate a minimal fix. Metric: repair success rate. Supported by the recovery benchmark (\Cref{sec:res-recovery-bench}) and oracle repair (\Cref{sec:res-causal}).
\end{itemize}

\textbf{Evaluation protocol.} We provide a train/dev/test split by task ID (198/42/43 tasks) to prevent data leakage. For Task C (four-class: runtime vs.\ output\_mismatch vs.\ answer\_error vs.\ analytical\_plan), \Cref{tab:baselines-classification} reports baselines evaluated on both the auto-labeled test split (194 traces) and the 47-trace human gold set. On the auto-labeled test split, the simple baselines (majority 61.6\%, silent heuristic 80.3\%) use heuristic rules, and ML baselines---TF-IDF + Logistic Regression (86.1\%), trace features + Random Forest (88.7\%), and code-aware + Logistic Regression (89.2\%)---are trained on the train split (1{,}101 traces, auto-labeled). Crucially, we also evaluate all baselines on the 47-trace human gold set to test whether high auto-split accuracy reflects genuine failure-stage prediction or merely imitation of the auto-labeling rules. On the gold set, the v2 classifier (with \texttt{gt\_path} comparison) achieves 76.6\% accuracy and 51.3\% macro-F1, up from 57.4\% for the old rule-based system. The silent heuristic achieves 95.7\% accuracy on the binary silent-vs-loud subtask, confirming that the silent/loud distinction is highly reliable. No baseline detects \texttt{answer\_error} (2 gold cases), as this stage requires external ground-truth checking unavailable at inference time. For Task D (step localization), the random-step baseline achieves 82.1\% and the last-step baseline achieves 100\% (mean trace length 1.69 steps), indicating the task is currently degenerate; step-localization benchmarking requires longer, multi-step trajectories.
